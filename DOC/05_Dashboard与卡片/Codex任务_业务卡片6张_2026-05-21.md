## 1. 第 1 批：Hermes truth_intake 入口（§4.1 + §4.2）

这两张是 Hermes truth_intake 流程的核心入口卡。Claude 在 `data/plugins/hermes_bridge/hermes_bridge.py` 已经有桥接逻辑，但 truth_intake state machine 还没建。**你只负责卡片函数本身**，state machine 串联留给后续任务。

### 1.1 真实性资料请求卡 `build_truth_intake_request_card`

**触发场景**：员工提出公司真实任务但没提供真实依据（公告 / 客户邮件 / 品牌文案 / 项目分析 / Hermes 深度分析 / 媒体生成 brief）。

**函数签名**：

```python
def build_truth_intake_request_card(
    *,
    task_title: str,                # 例："草拟端午客户问候邮件"
    task_id: str,                   # Harness task_id（如 abc12345）
    missing_fields: list[str],      # 例：["背景", "目标对象", "原文/附件"]
    task_brief: str = "",           # 员工原话（缩略到 200 字以内）
    accept_hints: list[str] = None, # 默认 ["飞书链接", "附件", "截图", "粘贴原文"]
) -> dict[str, Any]
```

**字段映射（来自 design.md §4.1）**：

| design.md 字段 | 实现 |
|---|---|
| 标题 | header.title = "📋 需要一点真实资料" |
| 温柔提示 | body 第一段 markdown，固定文案："我先帮你把这件事稳住。为了不让内容失真，我还需要一点可核验的资料。" |
| 需要补充 | 列 missing_fields，每项前加 `- ` |
| Harness 状态 | "⏸ `truth_intake / blocked`" |
| task_id | "`#" + task_id + "`" |
| 操作建议 | 列 accept_hints |

**header 颜色**：`orange`（需要补充）

**按钮**：底部一个 primary 按钮「📎 我现在补资料」（value = `{"action": "noop"}`，纯提示作用，不实际触发交互）

**收尾文案**（notation 字号）："有多少给多少，我会接着往下处理。"

### 1.2 资料已接收卡 `build_truth_intake_received_card`

**触发场景**：员工补充真实材料后（飞书链接 / 附件 / 文字粘贴），由 truth_intake 自动推一张确认卡。

**函数签名**：

```python
def build_truth_intake_received_card(
    *,
    task_id: str,
    sources_summary: dict[str, int],   # 例：{"附件": 2, "飞书链接": 1, "文字材料": 1}
    archive_path: str,                 # 例："data/harness_intake/raw/abc12345/"
    next_route: str = "Router",        # "Router" / "Hermes" / "Claude CLI"
) -> dict[str, Any]
```

**字段映射（来自 design.md §4.2）**：

| design.md 字段 | 实现 |
|---|---|
| 标题 | header.title = "✅ 资料已收到" |
| 状态 | "📦 已归档，正在继续处理" |
| 资料来源 | 拼 sources_summary，例："附件 2 个 · 飞书链接 1 个 · 文字材料 1 段" |
| 归档位置 | "`" + archive_path + "`" |
| 下一步 | "已交回 " + next_route + " 继续执行" |

**header 颜色**：`green`（正常完成）

**无按钮**（状态卡，纯展示）

## 2. 第 2 批：4 张数据 / 状态卡

### 2.1 来源追踪卡 `build_source_trace_card`

**触发场景**：所有公司事实类输出之后，跟随主回复一起推一张「这次回答基于什么」的卡。

**函数签名**：

```python
def build_source_trace_card(
    *,
    truth_status: str,                  # "已核验" / "部分待确认" / "需要补充" / "示例/假设"
    sources: list[str],                 # 例：["飞书 wiki: xxx", "附件: 端午方案.pdf", "知识库: 品牌规范"]
    kb_names: list[str] = None,         # 例：["品牌规范", "营销素材"]，None 时不展示
    unconfirmed: list[str] = None,      # 例：["预算", "发布时间", "客户审批意见"]
    risk_note: str = "",                # 例："以下内容仍需老板确认"
) -> dict[str, Any]
```

**字段映射（来自 design.md §4.3）**：5 个区块，按出现顺序：真实性状态 → 来源 → 知识库 → 未确认项 → 风险提示。每个区块都用 markdown，未确认项用 `- ` 列表。

**header 颜色**：按 truth_status 映射：
- 已核验 → `green`
- 部分待确认 → `orange`
- 需要补充 → `orange`
- 示例/假设 → `violet`

**header 标题**：固定 "🔍 来源追踪"

**无按钮**

### 2.2 DevOps 状态卡 `build_devops_status_card`

**触发场景**：技术小助手（巅池-技术（DevOps））收到查询指令时，由 builtin_commands / DevOps 插件触发。

**函数签名**：

```python
def build_devops_status_card(
    *,
    services: list[dict],              # 见下方 service dict 规范
    queue_summary: dict = None,        # 例：{"hermes": 2, "claude_cli": 0, "codex": 1}
    recent_errors: list[str] = None,   # 最多 3 条，超过截断
    next_action_hint: str = "",        # 例："Hermes 队列正常，无需干预"
) -> dict[str, Any]
```

**service dict 规范**：

```python
{
    "name": "AstrBot",                # 服务名
    "status": "正常",                  # "正常" / "异常" / "降级"
    "pid": "51997",                   # 可选
    "metric": "lark 3/3 在线",         # 关键指标，可选
}
```

**字段映射（来自 design.md §4.14）**：

| design.md 字段 | 实现 |
|---|---|
| 标题 | header.title = "🛠️ DevOps 状态" |
| 服务列表 | 拼一个 markdown 表格（服务 / 状态 / pid / 指标 4 列）|
| 关键指标 | queue_summary 拼成 "Hermes 队列：2 · Claude CLI：0 · Codex：1" |
| 错误摘要 | recent_errors 列前 3 条，每行带 ⚠️ 前缀 |
| 下一步 | next_action_hint，灰字 notation |

**header 颜色**：
- 全部 services.status == "正常" → `green`
- 任何 "异常" → `red`
- 任何 "降级"（无异常） → `orange`

**按钮**（可选，根据 next_action_hint 是否提示重启）：
- 如果 next_action_hint 包含「重启」字样 → 加一个 default 按钮「🔄 触发重启诊断」（value = `{"action": "devops_restart_prompt"}`）
- 否则无按钮

### 2.3 知识库沉淀卡 `build_kb_archive_card`

**触发场景**：原始材料归档并同步知识库后由 knowledge_base 模块推送。

**函数签名**：

```python
def build_kb_archive_card(
    *,
    archived_items: dict[str, int],     # 例：{"附件": 2, "飞书链接": 1, "文字材料": 1}
    kb_name: str,                       # 例："品牌规范" / "营销素材" / "中台运营"
    status: str,                        # "已入库" / "待审核" / "入库失败"
    archive_path: str,                  # 例："data/harness_intake/archived/xxx/"
    doc_id: str = "",                   # 知识库内部 doc id，可选
    error_hint: str = "",               # 入库失败时的诊断
) -> dict[str, Any]
```

**字段映射（来自 design.md §4.15）**：

| design.md 字段 | 实现 |
|---|---|
| 标题 | "📚 资料已沉淀" |
| 知识库 | "📦 入库到：" + kb_name |
| 来源 | archived_items 拼接 |
| 状态 | 按 status 显示带 emoji（✅ 已入库 / 🕐 待审核 / ❌ 入库失败）|
| 可追溯路径 | "`" + archive_path + "`" + (doc_id 非空时显示 "doc_id: `xxx`") |

**header 颜色**：
- 已入库 → `green`
- 待审核 → `orange`
- 入库失败 → `red`（同时显示 error_hint）

**无按钮**

### 2.4 员工待确认卡 `build_employee_pending_card`

**触发场景**：业务任务（邮件草稿 / 方案 / 文案）已经生成主体内容但有字段未补全，需要员工确认。

**函数签名**：

```python
def build_employee_pending_card(
    *,
    task_title: str,                    # 例："端午客户问候邮件草稿"
    pending_items: list[dict],          # 见下方 pending_item 规范
    current_draft_summary: str = "",    # 当前已生成内容的摘要（200 字内）
    task_id: str = "",                  # Harness task_id，可选
) -> dict[str, Any]
```

**pending_item dict 规范**：

```python
{
    "field": "发布时间",        # 字段名
    "hint": "默认 6 月 17 日，需确认",  # 可选默认值提示
}
```

**字段映射（来自 design.md §4.17）**：

| design.md 字段 | 实现 |
|---|---|
| 标题 | "🤔 还差一点信息" |
| 温柔提示 | 固定第一段："我已经先整理到这一步了，还需要你确认几项 —— 给我答案我就接着往下做。" |
| 待确认项 | pending_items 列表，每项 `- **field**：hint` |
| 当前可用内容 | current_draft_summary（如果非空），加 hr 分割 |
| 下一步 | "补充后我会自动继续生成"（notation 字号）|

**header 颜色**：`orange`（需要补充）

**按钮**（一个 primary）：「✏️ 我现在补」（value = `{"action": "noop"}`，纯提示，员工直接发文字回复即可）

### 3.3 风格一致性检查

每张卡都必须：

- [ ] header.template 颜色按 §0.1 规范 + §5（design.md）映射
- [ ] header.title 用 emoji + 短词（不超过 12 字）
- [ ] 第一段 body 是 heading_2 字号的标题
- [ ] markdown 用 `**xxx**` 加粗关键字
- [ ] 字段列表统一用 `- ` 前缀
- [ ] 路径 / id 用 `` ` `` 反引号包裹
- [ ] 结尾用 notation 字号 + 温柔提示语
- [ ] 不出现 "信息不足，请补全" 这种审计/质问语气（参考 design.md §1.2）

### 3.4 不要做的事

- ❌ 不要新增依赖（只用 stdlib + 现有 typing）
- ❌ 不要改 helper（`_button_row` / `_md_blocks_from_text`）
- ❌ 不要写 placeholder 数据生成 mock（这是纯 builder，调用方传什么就显示什么）
- ❌ 不要在卡片里硬编码任何业务 secret / open_id / chat_id
