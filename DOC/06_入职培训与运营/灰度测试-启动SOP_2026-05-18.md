> **不是**：给员工看的群规说明（那个在 [docs/grey_test_video/DC-Agent灰度测试操作指南.md](../docs/grey_test_video/DC-Agent灰度测试操作指南.md)）

## 一、灰度测试的真正核心 · 一句话

> **测试 "老板对 AstrBot 方案不满意 → 触发 Hermes 深挖 → 群内讨论补充 → 持续深挖 → 直到满意 → 沉淀" 这条完整闭环**，能不能在真实群聊压力下跑通。

**不是**收集对话样本。**不是**让员工自由测试 AI 助手。是测一个**具体业务闭环**。

---

## 二、跟今天 AI 助手内测的区分

| | 灰度测试 | AI 助手内测（5/18 启动）|
|---|---|---|
| **测试目标** | Hermes 深挖闭环 + 群讨论压力 | AI 助手日常工作可用性 |
| **场景设定** | 老板拿出一个"不满意的方案"作为起点 | 员工自由提问 |
| **主角机器人** | 巅池-推广 01 ⬅️ 主动发言 | 巅池-Agent 小助手 |
| **辅助机器人** | 巅池-技术（DevOps）⬅️ 静默观察 | 无 |
| **群环境** | 专门拉的"灰度测试群"（不是真实业务群！）| 专门拉的"AI 小助手内测群" |
| **员工知情** | 看视频 + 卡片 + 群规明示 | 群规告知 |
| **测试时长** | 1 次性（按视频和卡片走完一轮）| 一周持续 |

**两个测试可以独立跑，互不干扰**，但**别混在同一个群**。

## 四、启动前 · 必做合规

灰度测试虽然是**专门拉测试群**（而非潜入真实业务群），但仍然涉及"记录员工对话"，需要：

### 4.1 群成员事先知情

由于群是新建的、员工自愿加入、群规明示有静默观察机器人，**合规风险比"潜入真实群"低很多**。但仍然建议：
- 在群规中明确"巅池-技术（DevOps）会静默记录系统体验，仅用于产品改进"
- 飞书卡片（`feishu_grey_test_card.json`）里已经讲清楚，**进群第一条发它**即可

### 4.2 数据使用承诺

- 灰度记录的对话**仅供产品研发使用**
- 不对外、不商用、不上传第三方
- 测试结束后归档 + 通知员工

## 五、启动步骤（按顺序跑）

### Step 1 · 预检查（5 分钟）

```bash
cd /Users/dianchi/DC-Agent
bash scripts-tools/verify_llm_after_restart.sh
```

确认输出 "🎉 全部通过"。

### Step 2 · 确认 silent_observer plugin 状态

```bash
grep -A2 'Loading plugin silent_observer' astrbot.log | tail -3
```

应该看到 plugin 已加载。默认配置：监听 `巅池-技术`、`enabled: true`。

### Step 3 · 建灰度测试群

1. 飞书新建群 → 命名 **「DC-Agent · 灰度压力测试群」**
2. 拉 **6-8 个核心员工**：
   - 老板 1 人
   - 部门负责人 2-3 人（市场部 / 品宣部 / 策略部 各 1 个）
   - 普通员工 2-3 人（执行运营 / 影视部 / 市场部 各 1 个）
   - 你 1 人（群管理 + 应急响应）
3. **同时拉 2 个机器人**进群：
   - 巅池-推广 01（互动主角）
   - 巅池-技术（DevOps）（静默观察）

### Step 4 · 群内启动（按视频里的脚本走）

1. **发飞书卡片**：把 [feishu_grey_test_card.json](../docs/grey_test_video/feishu_grey_test_card.json) 的内容发到群里（飞书后台 → 消息卡片 → 粘贴 JSON → 发送）
2. **发介绍视频**：把 `DC-Agent完整版同步版.mp4` 上传到群里（13MB · 4 分钟），让员工看
3. **群禁言 10 分钟**：让员工有时间看视频 + 阅读卡片 + 改群名片
4. **10 分钟后解禁**，发"请各位回复 `1` 或 `到` 完成点名"
5. 所有人点名完毕后，**老板开场**（按视频里的脚本）：
   > "上一版 AstrBot 方案我不满意，问题是 XXX。希望 Hermes 围绕 YYY 继续深挖。"
6. 部门负责人 + 员工开始**真实补充约束**（预算、人手、时间、客户问题）
7. 巅池-推广 01 整理 + 触发深挖
8. 持续迭代直到老板说"基本满意"

### Step 5 · 全程在旁监控（你的工作）

| 监控项 | 怎么看 |
|---|---|
| AstrBot 是否实时响应 | 看群里巅池-推广 01 回复有没有延迟 |
| Hermes 是否正确触发 | `tail -f hermes-config/logs/gateway.log` |
| 巅池-技术 是否在静默记录 | `ls -lh data/plugin_data/silent_observer/$(date +%Y-%m-%d).jsonl` 看文件大小是否在长 |
| 错误日志 | `tail -f astrbot.log \| grep -E "ERROR\|WARN"` |
| LLM 消耗 | 登录 aihubmix 后台看 dianchi key 当前消耗 |

## 六、监控信号 · 测试是否有效

按原版操作指南第 8 节，**测试有效** 的标志：

- ✅ 老板的不满意原因被说清楚
- ✅ 部门负责人和员工的限制条件被巅池-推广 01 记录
- ✅ 方案从"泛泛而谈"变得更具体
- ✅ 系统能连续承接多轮讨论，没有忘记核心问题
- ✅ 不满意时能继续深挖，而不是直接结束
- ✅ 最后能形成阶段性结论 + 执行建议 + 问题清单

**测试无效** 的信号：

- ❌ 巅池-推广 01 回复跟上下文脱节
- ❌ 多人讨论后系统只记得最后一个人的话
- ❌ "继续深挖"指令不触发 Hermes
- ❌ Hermes 跑出的结果回不到群里
- ❌ 群活跃度低，员工不愿真实表达（看视频白看了）

## 七、灰度结束后

### 7.1 数据归档

```bash
TS=$(date +%Y%m%d_%H%M%S)
mv data/plugin_data/silent_observer "data/_archive_silent_observer_$TS"
mkdir -p data/plugin_data/silent_observer
echo "✅ 灰度数据归档到 data/_archive_silent_observer_$TS"
```

### 7.2 整理复盘报告（给老板）

按原版操作指南第 9 节，沉淀这些内容到 `DOC/灰度测试-复盘报告_<日期>.md`：

1. 本次老板不满意的原始原因
2. 群内补充的业务约束和真实反馈
3. Hermes 深挖后的阶段性结论
4. 最终可执行方案 or 仍需补充的问题
5. 员工和部门负责人对系统体验的意见
6. 技术链路问题、权限问题、消息识别问题、异常记录

### 7.3 通知员工

正式飞书消息告知所有参与员工：
- 测试已结束
- 数据已归档
- 后续会基于这次反馈持续打磨产品

## 八、应急停服

### 8.1 巅池-推广 01 答错关键问题 / 不响应

```bash
# 一键重启 AstrBot
launchctl kickstart -k gui/$(id -u)/io.astrbot.bot
sleep 30
bash scripts-tools/verify_llm_after_restart.sh
```

### 8.2 完全下线灰度（紧急）

1. **飞书后台**：把巅池-推广 01 和巅池-技术从灰度群移除
2. **群内通知**：群里发"系统遇到问题，灰度测试暂停，稍后通知"
3. **数据归档**：参考 7.1
4. **复盘事故**：写一份 incident report 到 `data/watchdog/incidents/grayscale-<日期>.md`

### 8.3 单员工要求删除自己的数据

参考 [AstrBot应急回滚手册](./AstrBot应急回滚手册_2026-05-15.md) 第 5 节。
