# 卡片系统工程级升级与验收方案

## 目标

卡片系统不再只是一组模板文件，而是一套可注册、可检测、可灰度、可回滚、可追踪的工程闭环。

核心要求：

- 每张卡必须有固定 `card_type`、版本、负责人、builder、触发条件和兜底策略。
- 任意模板升级前先通过本地健康检查，再进入飞书真机灰度。
- 任意测试成功后必须补上下一步衔接，不能只停在“通过”。
- 闲聊、任务等待、长回复、业务卡、入职、培训、培训测试、Skill 运维、生图、生视频都必须从同一套卡片系统登记和自检。

## 触发判定原则

运行时网关：

- 插件发卡必须优先走 `dc_engines.card_runtime.send_card_via_runtime()`。
- 插件 finalize 必须优先走 `dc_engines.card_runtime.finalize_card_via_runtime()`。
- 网关负责 `card_type` 注册校验、飞书发送、失败兜底和运行事件记录。
- `scripts-tools/card-grey-push.py` 只是灰度入口，不是独立实现；它也必须调用运行时网关。
- 新业务插件如果直接调用 `streamer.start()`，必须同步补运行事件、健康检查和合同说明，否则视为不合格。

闲聊卡：

- `intent=casual` 必须走闲聊卡。
- `intent` 为空或 fallback，且原始消息短、没有任务动词时，走闲聊卡。
- 闲聊卡发送失败时，不清空原回复，保留纯文本兜底。

等待卡：

- 明确任务类请求触发等待卡。
- 有 `reasoning_tier` 时触发等待卡。
- `intent=casual` 不触发等待卡。
- 等待卡发送失败时，不阻断 LLM 正常回复。

长回复卡：

- 有等待卡时，最终结果覆盖等待卡。
- 没有等待卡时，长结构化内容才转结果卡。
- finalize 失败时保留纯文本兜底。

业务卡：

- 业务插件不得绕过 `card_system.py` 的注册表。
- 新卡上线前必须补齐样卡 payload，让健康检查能生成样卡。
- 危险按钮必须有确认卡或权限判断，不允许直接执行。

## 升级流程

1. 在 `templates.py` 新增或调整 builder。
2. 在 `card_system.py` 登记 `CardSpec`，补齐：
   - `card_type`
   - `version`
   - `owner`
   - `builder`
   - `triggers`
   - `fallback`
3. 在 `_sample_payload()` 增加样卡数据。
4. 运行健康检查：

```bash
scripts-tools/card-system-health.py
```

5. 设置灰度版本：

```bash
scripts-tools/card-system-health.py --set-version <card_type> <version> --rollout grey --updated-by codex --note "<change note>"
```

6. 重启主聊天并自动跑工程自检：

```bash
scripts-tools/safe_restart.sh astrbot
```

7. 按 `--next-step` 输出继续真机验证。

## 回滚流程

查看版本：

```bash
scripts-tools/card-system-health.py --versions
```

回滚单张卡：

```bash
scripts-tools/card-system-health.py --rollback <card_type> --updated-by codex --note "<rollback reason>"
```

回滚后必须继续：

```bash
scripts-tools/card-system-health.py
scripts-tools/safe_restart.sh astrbot
```

如果自检通过，再发一张对应真实卡验证；如果自检失败，停止灰度。

## 测试后的衔接规则

任何验证通过后，都必须明确下一环：

- 本地健康检查通过：继续重启后自检。
- 重启后自检通过：继续发真实飞书卡。
- 真实闲聊卡通过：继续任务等待卡。
- 真实任务等待卡通过：继续 finalize 结果卡。
- 真实业务卡通过：继续按钮动作、权限和回滚验证。
- 灰度通过：再把版本状态改为 `stable`。
- 任意失败：保留纯文本兜底，记录运行事件，先修复失败再继续。

命令：

```bash
scripts-tools/card-system-health.py --next-step
```

## 真机灰度验收

最小灰度集：

- 闲聊：短句问候，确认闲聊卡出现。
- 任务：例如“帮我写一封端午客户邮件”，确认等待卡出现、时间进度正常、最终结果覆盖。
- 长回复：结构化内容确认标题、正文、代码框、表格、加粗、颜色。
- 入职：新好友或人工推送，确认部门/职能按钮排版。
- 培训：课程卡、测试题卡、按钮居中和字号。
- 生图/生视频：等待状态、进度和最终状态。
- Skill 运维：列表、审阅、危险确认、权限拒绝。

验收后查看运行事件：

```bash
scripts-tools/card-system-health.py --events --limit 20
```

给指定接收人发真实灰度卡：

```bash
scripts-tools/card-grey-push.py \
  --receive-id-type union_id \
  --receive-id <receiver> \
  --card-type casual_reply \
  --card-type thinking_waiting
```

## 未来升级方式

小改：

- 只改颜色、字号、间距、按钮样式。
- 保持 `card_type` 不变，版本号小幅递增。
- 走本地健康检查、重启后自检、两张真机卡。

中改：

- 改布局结构、字段顺序、按钮组、等待进度表现。
- 版本进入 `grey`。
- 至少覆盖闲聊、等待、结果、一个业务卡真机验证。

大改：

- 新增卡片类型、新增插件 owner、新增危险动作或新增触发判定。
- 必须补 `CardSpec`、样卡、运行事件、回滚路径。
- 先灰度给内部小范围，再改为 stable。

系统级升级：

- 模板注册表和触发判断继续放在 `card_system.py`。
- 插件只负责业务输入和发送，不私自复制触发规则。
- 安全重启必须保留卡片系统自检。
- 失败时永远有纯文本兜底，不能让用户看到空响应。

## 当前结论

当前系统已经从“模板集合”升级为“工程化卡片运行层”：

- 有注册表。
- 有版本状态。
- 有健康检查。
- 有重启后自检。
- 有运行事件。
- 有下一步衔接出口。
- 有灰度和回滚入口。

下一阶段重点不是再堆模板，而是真机灰度覆盖更多业务触发路径，并把每次成功后的衔接固定执行。
