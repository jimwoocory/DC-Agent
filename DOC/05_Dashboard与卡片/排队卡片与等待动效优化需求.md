# 排队卡片与等待动效优化需求

## 需求背景

巅池-Agent 小助手接入 `agy / Antigravity` 后，需要在高峰时对请求做真实排队。当前 router 已能计算真实队列位置和预计等待时间，但前端卡片还需要更清晰地表达等待状态，并提供备用 AI 方案入口，避免员工误以为系统卡住。

本需求面向“小助手排队卡片 / 等待动效 / 备用方案选择”模块。

## 核心目标

- 排队信息必须真实、清楚、人性化。
- `当前位置` 和 `预计等待` 必须来自后端真实队列数据，不能写死。
- 员工排队时可以选择继续等待，也可以选择备用 AI 方案免排队。
- 员工侧不展示 OAuth、验证码、登录态、provider、AIHubMix 等技术词。

## 命名约定

- 主通道名称：`巅巅小助手`
- 备用通道名称：`池池小助手`

## 卡片文案

卡片标题：

```text
小助手排队中
```

主文案：

```text
哎呀，现在小助手太忙啦，当前同时接待人数已经超过 9 人。
我已经帮您排好队了，请您耐心等待⌛️
```

状态字段：

```text
当前位置：第 {queue_position} 位
预计等待：{eta_text}
当前通道：巅巅小助手
```

备用入口按钮：

```text
不想排队，使用池池小助手
```

点击备用入口后的提示：

```text
好的，我马上请池池小助手先帮您处理～
```

## 交互要求

1. `queue_position` 必须来自后端真实队列位置。
2. `eta_text` 必须来自后端真实 ETA。
3. 卡片需要有等待动效，例如轻量 loading、进度条、呼吸点或计时刷新。
4. 卡片每 10-15 秒自动刷新一次等待状态。
5. 员工点击“不想排队，使用池池小助手”后：
   - 取消当前 `agy` 队列任务；
   - 切换到备用 provider：`aihubmix/gemini-3-flash-preview`；
   - 使用原始用户消息继续回复；
   - 卡片更新为“池池小助手处理中”。

## 后端数据字段

卡片渲染需要后端提供以下字段：

```json
{
  "job_id": "真实队列任务 ID",
  "queue_position": 3,
  "eta_at": 1710000000,
  "eta_text": "约 1 分钟",
  "primary_channel_name": "巅巅小助手",
  "fallback_channel_name": "池池小助手",
  "fallback_provider_id": "aihubmix/gemini-3-flash-preview",
  "original_prompt": "用户原始消息"
}
```

## 异常处理

如果 `agy` 出现 OAuth、验证码、超时、登录态异常，不向员工展示技术原因。

员工侧统一展示：

```text
小助手当前接待人数较多，您可以继续等待，或使用池池小助手先处理。
```

如果备用 AI 也不可用，展示温和失败提示，并保留队列任务：

```text
池池小助手现在也有点忙，我先继续帮您保留排队位置。
```

## 验收标准

- 12 人同时请求时，前 9 个按 `agy / 巅巅小助手` 正常处理，后续进入排队卡片。
- 卡片中的排队人数和等待时间与后端 QuotaGate 数据一致。
- 点击备用按钮后，不再等待 `agy`，由 `池池小助手` 立即处理。
- 员工不会看到 OAuth、验证码、登录态、provider、AIHubMix 等技术词。
- 卡片视觉上清楚、有等待感，不像系统无响应。
