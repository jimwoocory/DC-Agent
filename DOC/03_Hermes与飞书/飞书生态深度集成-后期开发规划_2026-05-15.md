### 1.2 下午暴露的两个核心问题

**问题 A · 飞书的"个人云盘私有"设计**

- 公司云文档总量 503GB（云文档 434GB + 聊天 59GB + 视频会议 9.6GB）
- 但超级管理员能看到的只有"自己上传的 + 别人主动分享的"
- 飞书故意挡住"管理员强制访问员工私人文档"的 API，连付费的"文件管理 增值版"也只支持转移所有权，**不支持下载内容**

**问题 B · 公司真正的痛点：防员工把数据私有化**

- 员工把客户名单、营销资料、项目文档存到**飞书个人云盘**
- 离职时带走，公司损失资产
- 这是**企业管理诉求**，不是"窥探员工隐私"

### 1.3 浮出的新方向 · 妙搭

- 飞书官方妙搭（Aily）：低代码 + AI 平台，业务部门能自己搭工具
- 影视飓风等公司有过实践分享
- 跟 DC-Agent 是 **互补**（妙搭做前台、DC-Agent 做后端），不是替代

## 三、三大新方向（后期主要开发任务）

### 方向 1：防数据私有化工具链（管理需求 · 最紧急）

公司核心痛点。3 件事必须做，从易到难：

#### 1.1 离职交接自动备份（P0 · 4-6 小时工作量）

**目标**：员工离职流程触发后，小助手自动把该员工飞书云盘文件备份到 NAS `/离职员工归档/<姓名>_<离职日期>/`

**实现**：
- 开权限：`admin:cloud_disk:transfer` 或飞书的"员工离职"事件订阅
- 写 plugin：`data/plugins/offboarding_backup/`
  - 监听 `employee.left` 事件（飞书 webhook）
  - 调离职交接 API 拉员工的文件列表
  - 用 feishu_hub 走当前的 download 逻辑批量下到 NAS
- 看门狗加 `offboarding_event_count` 计数器，验证有事件就处理了

**业务效果**：员工离职那天，你**自动**收到一封通知"X 员工的 N 个文件已备份到 NAS /离职归档/"。

**合规前提**：员工合同里必须有"工作产出归公司所有"条款，否则就是非法侵权。

#### 1.2 异常下载告警（P0 · 3-4 小时）

**目标**：有人短时间下载大量公司文档 → 实时告警老板

**实现**：
- 开权限：`admin:audit_log:readonly`
- 写 daemon：`scripts-watchdog/feishu_audit_monitor.py`
  - 每 5 分钟拉一次审计日志（仅"下载/导出/分享外部"动作）
  - 规则：单员工 1 小时内下载 > 50 个文档 → 触发告警
  - 通过飞书"巅池-Agent小助手"私聊推给老板
- launchd 装上 `com.dcagent.feishu-audit.plist`

**业务效果**：员工离职前如果想"批量备份带走"，5 分钟内你就知道了。

#### 1.3 外发管控 + 水印（P0 · 30 分钟，纯配置）

**实现**：纯后台配置，无代码

- 飞书管理后台 → 安全 → 外发管控 → 设白名单（仅 @公司域 / 客户白名单可分享）
- 飞书管理后台 → 安全 → 水印 → 全员强制（员工自己截屏外泄也能追溯）

### 方向 2：妙搭集成（业务前台扩展）

#### 2.1 妙搭定位

妙搭做"业务部门天天用的轻量前台"，DC-Agent 做"重活后端"。两者**通过飞书 API 互通**。

#### 2.2 四条扩展路径（按工作量从小到大）

**路径 A · 妙搭 Webhook → DC-Agent 后端**

- 妙搭里员工点按钮 → 触发 HTTP Webhook → 打到 Hermes Gateway `:8644/webhooks/lark_action`
- 用法举例：员工在"客户跟进表"点"AI 分析风险"按钮 → DC-Agent 跑深度任务 → 结果写回多维表格"风险评级"字段
- **工作量**：2-3 小时（每个新动作 30 分钟）
- **价值**：让员工觉得"在飞书里做事，AI 自动帮忙"

**路径 B · DC-Agent 把结果写回多维表格**

- DC-Agent 通过 `bitable.v1.app_table_record.*` API 主动写飞书多维表格
- 用法举例：群消息扫到客户提了新需求 → 自动 INSERT 到"客户需求表"
- **工作量**：4-6 小时（一次性建立 bitable 写入 SDK），后续每个表 30 分钟
- **价值**：员工在飞书原生界面就能看到 AI 沉淀的内容

**路径 C · 妙搭 AI 字段用我们自己的 LLM**

- 妙搭"AI 字段"支持自定义 LLM provider（OpenRouter / Aihubmix）
- 配你的 API key → 妙搭 AI 字段实际跑你的模型
- **工作量**：1 小时（配置而已）
- **价值**：模型可控 + 成本可控 + 跨模型试

**路径 D · 妙搭应用嵌进 DC-Agent dashboard**

- 妙搭应用本身是 iframe / 嵌入式的
- 塞进 AstrBot Dashboard 当 plugin page（跟今天做的 system_entries 同机制）
- **工作量**：30 分钟
- **价值**：业务部门搭的小工具统一在一个入口管

#### 2.3 妙搭集成的推荐顺序

C → A → B → D。先体验（C 配 LLM），再让员工接入（A Webhook），再做反向写入（B），最后做整合视图（D）。

## 五、合规 / 风险

### 5.1 法律风险

- **员工告知书必须先签**：所有"读员工文件 / 聊天 / 打卡"的功能上线前，员工必须明确知情同意
- **离职交接也要写进合同**：否则离职员工可以告"侵犯隐私"
- **审计日志使用边界**：审计日志告警 ≠ 监控员工，写规则只针对"异常行为"（数量级、对外分享），不要拿来看员工日常工作内容
- **敏感数据隔离**：客户身份证号、银行卡号等绝对不进 AI 知识库（即使是公司资产）

### 5.2 技术风险

- **`drive:drive:readonly` 已开**：app_secret 一旦泄漏，全公司云盘内容外流。必须把 secret 严格保护在 `data/feishu_whitelist.yaml`，不要 commit 进 git，不要传给第三方
- **NAS 是单点**：所有同步内容都到 NAS，NAS 挂了 = 全完蛋。需要：定期 rsync 到第二块硬盘 / 异地备份
- **飞书 API 限流**：审计日志 / 离职交接是低频 API，限制宽松。但群消息扫描 / OKR 同步是高频，要走 hub 的统计 + 限流

### 5.3 业务风险

- **员工反感**：员工知道工作内容被 AI 读，可能反弹。处理：**只读"工作产出"，不读"日常聊天"**，前者归公司，后者归员工
- **过度依赖飞书**：所有都建在飞书上，飞书产品策略变了（涨价、停服、限流）就被动。处理：**核心资产沉淀到 NAS / Git，飞书只是入口**
- **AI 幻觉**：LLM 读了不准确的资料可能瞎编。处理：**关键决策 / 客户报价不能让 AI 自动决定**，AI 只做提示
