## 三、Week 1 基础数据流水线（4-5 天 / ~18 小时）

| Day | 功能 | 工时 | 产出 |
|---|---|---|---|
| D1-2 | **群消息抓取 daemon**（im.message.list 持续拉）| 6h | `messages.db` 有数据持续流入 |
| D3 | **链接提取**（消息正文里飞书内文档 URL → 自动入 kb）| 3h | 飞书云相互分享的文档自动入知识库 |
| D4 | **外部 URL 抓取**（playwright / firecrawl 抓非飞书链接）| 6h | 团队群里贴的 GitHub / 文章自动入 kb |
| D5 | **dc-watchdog 加飞书业务探针**（消息库增长率、链接抓取失败率）| 3h | 看门狗能告警业务异常 |

**业务上你拿到**：
- 群里说过啥能查回来
- 团队分享的外部资源全在一个"链接库"
- 飞书内文档自动同步知识库（不用每个人手动复制）

## 八、依赖与风险

| 项 | 风险 | 应对 |
|---|---|---|
| **ops 配飞书凭证** | 没配 → 整张表全部 blocked | Week 0 之前必须搞定，催 ops |
| **NAS 在线** | NAS 192.168.1.35 现在离线 | 上线 NAS / 改用本地存储 |
| **LLM token 成本** | 群消息 + 会议纪要 + 决策识别都要喂 LLM | 先过滤关键词命中再喂，预算每月 ¥500-2000 |
| **合规通知** | 员工聊天被存有隐私风险 | Week 1 启动前发员工通知 + 提供退出 |
| **飞书 API 限流** | 历史消息拉取慢 | 后台 daemon 慢慢拉，不抢业务流量 |
| **存储膨胀** | 1 个 50 人群一天可能几百条消息，全存知识库 N 个月几 GB | 用"自动摘要 + 保留原文索引"，不全文进 kb |

## 九、产品决策点

| 点 | 决策建议 |
|---|---|
| **要不要全做？** | 至少 Week 1-2（基础 + HR），Week 3-4 看公司预算决定 |
| **顺序能调吗？** | Week 1 必须先（其他都依赖它的数据流水线）；Week 2-4 顺序可对调 |
| **要不要等飞书凭证全配齐再启动？** | 不用——基础设施代码可以现在写，等凭证齐了热插拔进去 |
| **MVP 拿来给老板演示选哪个？** | "周报自动生成" + "决策审计" + "客户对话画像"——产品感最强 |

## 十、SLI vs 业务数据（澄清）

容易混淆的两件事，写清楚：

| | SLI（监测我们的系统）| 业务数据（飞书云里你公司的内容）|
|---|---|---|
| **看的是** | 飞书 API 调了多少次、失败几次、平均多快 | 营销方案 PDF 第 3 页内容、客户提了啥需求 |
| **谁用** | watchdog / dashboard | 业务 plugin / 知识库 / 推荐 |
| **谁产生** | `feishu_hub.stats` 顺手记的 | 业务 plugin 主动调飞书 API 爬下来 |

**这张表里所有的"业务功能"都是后者**（爬业务数据）。SLI 是顺带产物，免费拿到。
