## 2. 我的卡点（agy 调不通）

`scripts-tools/dianchi-tech-night.sh` 阶段 A 执行：

```bash
agy -p "<prompt>" --dangerously-skip-permissions --print-timeout 25m
```

结果：

```
Authentication required. Please visit the URL to log in:
  https://accounts.google.com/o/oauth2/auth?...
Waiting for authentication (timeout 30s)...
Error: authentication timed out.
```

排查到的根因：

1. agy token 存 `~/.gemini/oauth_creds.json`，里面有 `access_token` / `refresh_token` / `expiry_date`
2. 实测 `expiry_date` 显示 token 在今早 07:56 BJT 就过期了
3. **`agy -p`（headless print 模式）不会自动用 refresh_token 续期**，直接抛 OAuth URL 等人贴 code
4. 只有 `agy -i`（交互模式）会自动续期
5. 跟从哪起子进程无关（cron / 普通 shell / AstrBot 进程都一样），是 agy `-p` 自己的设计
6. 你之前在 `data/plugins/llm_router/cli_runner.py:97 run_antigravity()` 跑短任务能成，是因为那时候 token 还在有效期内

期间用户手动给过一次 OAuth code，我用 `echo "<code>" | agy -p ...` 喂进去，那一次能跑通，但**没写回 `oauth_creds.json`**，下次新进程依然失败。

## 5. Codex 判定（2026-05-21）

### 5.1 结论

本任务是 `dianchi_tech` 日报 cron 闭环，目标是每天凌晨无人值守地抓硅谷 AI 资讯、生成日报、09:00 推送飞书和写入 wiki。

**本任务继续以 agy/Gemini 作为阶段 A 主方案。Tavily + aihubmix Gemini 只作为兜底方案，不替代 agy。**

原因：

1. `agy/Gemini` 的强项正是搜索、跨来源理解、信息整理和分析，适合做日报阶段 A 的主引擎。
2. 日报的价值不只是抓 URL，而是让 Gemini 直接参与“搜索 + 归纳 + 判断重要性”。
3. cron 无人值守确实不能卡在 OAuth/验证码上，所以必须加健康检查、超时和兜底。
4. 正确架构是：**agy 主用；如果 agy 登录态/授权/超时异常，则自动降级 Tavily + aihubmix。**

### 5.1.1 闭环定位

`agy` 在本任务里只有两个职责：

1. **搜索**：围绕 OpenAI / Anthropic / Google Gemini / xAI Grok 抓取当天值得关注的 AI 动态。
2. **初步分析**：用 Gemini 的长上下文和多来源理解能力，先做新闻筛选、去重、归类和初步判断，产出 `raw_news.md`。

`agy` 不负责最终深度报告。

完整闭环如下：

```text
01:00 夜间任务
  → agy / Gemini 搜索 + 初步分析
  → raw_news.md
  → Claude CLI 深度分析 + AstrBot/Hermes 学习 + DC-Agent 只读巡检
  → report.md
  → 巅池-技术（DevOps）以维护者身份分发
  → 飞书私聊蔡挺
  → 飞书知识库『DC-Agent 运维』
  → Mac mini 本地 data/dianchi_tech/{DATE}/
  → NAS 知识库同步目录
```

也就是说：`agy` 是日报的数据入口和第一层智能筛选；Claude CLI 是深度分析和工程巡检；巅池-技术（DevOps）负责把结果沉淀到知识库、本地和 NAS。

### 5.1.2 agy 实测频率边界

Codex 在 2026-05-21 做过真实 `agy --print` 冒烟：

| 目标频率 | 结果 | 结论 |
|---|---|---|
| 6 RPM | 6/6 成功 | 稳定 |
| 9 RPM | 9/9 成功 | 稳定，建议生产安全值 |
| 12 RPM | 11/12 成功，1 次 auth_required | 风险边界 |
| 15 RPM | 7/15 成功，8 次 auth_required | 不适合作为稳定生产值 |
| 20 RPM | 16/20 成功，4 次 auth_required | 高风险 |

本日报任务不是高并发场景，每天 01:00 只需要阶段 A 跑一次。为了稳妥：

- 每日 `dianchi_tech` 阶段 A 只允许 **1 次 agy 主调用**。
- 单次 agy 超时建议 **180-300 秒**，不要 25 分钟长等。
- 发现 `Authentication required`、`authorization code`、`oauth`、验证码、timeout 时立即失败并走兜底。
- 不要在 cron 中循环重试 agy，避免触发更多授权异常。
- 如需重试，最多隔 60 秒重试 1 次；仍失败就走 Tavily + aihubmix。

### 5.2 三个方案的取舍

#### 方案 A：token 刷新守护

不建议。

原因：

- 涉及直接读写 Google OAuth token，维护风险高。
- 可能与 Google/Antigravity 的认证策略冲突。
- 即使短期可用，也容易在 MFA、账号策略、refresh token 轮换时再次失效。
- 不适合作为公司日报的稳定生产路径。

#### 方案 B：agy-bridge 本地服务

采用精简版，作为主方案实现方向。

说明：

- 不做复杂常驻 TUI，不包 `agy -i` 全屏交互。
- 做一个轻量 PTY wrapper 或本地 bridge，用 PTY 启动 `agy --print`。
- Codex 已实测：普通 PTY 下 `agy --print` 可以返回，HTTP bridge 探针也能返回。
- bridge/wrapper 必须检测 OAuth、authorization code、验证码、timeout，一旦发现立即失败并触发兜底。
- 日报 cron 阶段 A 调用 wrapper/bridge，而不是裸调 `agy -p`。

#### 方案 C：Tavily + aihubmix Gemini

保留为兜底。

这是本任务的备用方向：

- 当 agy 主通道不可用时，Tavily 负责抓取真实新闻来源和 URL。
- aihubmix `gemini-3-flash-preview` 负责整理成中文 markdown。
- Claude CLI 阶段 B 继续负责分析、学习、巡检和写报告。
- reporter.py 继续负责飞书 IM + wiki 写入。

### 5.3 与 router 中 agy 方案的边界

这里要区分两个完全不同的场景：

| 场景 | 是否使用 agy | 原因 |
|---|---|---|
| `dianchi_tech` 日报 cron | 主用 agy，失败降级 | Gemini 搜索/分析是核心价值，但 cron 要有兜底 |
| 巅池-Agent 小助手日常闲聊 | 可以低频使用 | 短任务、可排队、可快速切备用 |
| 小助手排队卡片 | 可以展示 agy 队列 | 由 QuotaGate 控制，失败可切备用 |
| 深度任务 / 日报深度分析 | 不使用 agy | agy 只做搜索和初步分析，深度报告交给 Claude CLI |

router 中的 `agy` 方案已经加了保护：

- 低频使用；
- 超时保护；
- QuotaGate 排队；
- 队列卡片；
- 可切备用通道。

这些保护可以借鉴，但日报 cron 需要更简单的单任务保护：一次主调用，失败立即走兜底。
