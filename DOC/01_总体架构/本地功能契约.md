# 本地功能契约（升级前必读）

> 记录 JW-Bot fork 在 upstream AstrBot 之上的本地添加点。每次升级 upstream 前先核对本表，确认改动文件中是否触及"上游钩子"——避免 git auto-merge 静默丢失本地代码。

最近更新：2026-05-12（AstrBot v4.22.3 → v4.24.2 升级后写入）

---

## 三个层次的本地修改

| 层次 | 风险 | 例子 |
|---|---|---|
| **A. 完全独立的本地文件** | 🟢 几乎零冲突 | `astrbot/core/harness/*`、`astrbot/core/router.py`、`astrbot/runtime/*`、`astrbot/shared/*` |
| **B. 上游文件里加新行** | 🟡 git 一般能 auto-merge | 大部分 import 语句、注册项 |
| **C. 上游 dataclass / 类签名里加字段** | 🔴 **git auto-merge 会静默丢失** | `MainAgentBuildConfig` 加 `lossless_*` 字段 |

**核心教训**：尽量把本地添加放到 A 层。C 层模式（在上游 dataclass 里加字段）是最危险的，今天踩了 2 次。

## 升级前 Checklist

1. **fetch upstream + 看 commit 范围**
   ```bash
   git fetch upstream --tags
   git log --oneline master..upstream/master | wc -l   # < 50 commits 是健康节奏
   ```

2. **看冲突预览**
   ```bash
   git checkout -b upgrade/vX.Y.Z
   git merge --no-commit --no-ff vX.Y.Z
   ```

3. **冲突分类**
   - 7 个 changelog + 字体 → 接受 upstream (`--theirs`)
   - .py 文件 → 看是不是 ruff 噪音
     - 是 ruff 噪音 → `--theirs`
     - 真有逻辑 → 手解

4. **关键文件改动后必查（C 层风险点）**
   ```bash
   # 本地的 lossless 字段还在吗？
   grep "lossless_context_enabled" astrbot/core/astr_main_agent.py

   # 本地的 harness 初始化还在吗？
   grep "self.harness_store" astrbot/core/core_lifecycle.py
   grep "from astrbot.core.harness import" astrbot/core/core_lifecycle.py
   ```

5. **冒烟测试**
   ```bash
   cd /Users/dianchi/DC-Agent
   /Users/dianchi/.local/bin/uv run python -c "
   from astrbot.core.config.default import VERSION
   from astrbot.core.initial_loader import InitialLoader
   from astrbot.core.harness.satisfaction import SatisfactionDetector
   print('✓ v' + VERSION, '/ harness OK')
   "
   ```

6. **重启服务（开发期可直接 launchctl）**
   ```bash
   launchctl unload ~/Library/LaunchAgents/io.astrbot.bot.plist
   launchctl load ~/Library/LaunchAgents/io.astrbot.bot.plist
   ```

7. **登录验证**
   ```bash
   curl -s -X POST http://localhost:6185/api/auth/login \
     -H "Content-Type: application/json" \
     -d '{"username":"Dianchi.boss","password":"<MD5>"}' | jq .status
   # 应返回 "ok"
   ```

## ruff 配置对齐（避免 80% 假冲突）

今天的合并 232 个文件改动里大多数是 ruff format 噪音（quote 风格、换行）。永久解决：让本地的 ruff 配置**完全等于** upstream 的 `pyproject.toml [tool.ruff]` 段。

```bash
diff <(awk '/^\[tool\.ruff/,/^\[/' pyproject.toml) \
     <(git show upstream/master:pyproject.toml | awk '/^\[tool\.ruff/,/^\[/')
```

若有差，对齐到 upstream。然后跑一次 `ruff format .` 入 commit。以后再跑 ruff 不会再产生 vs upstream 的差异。
