# Codex 工作流 SOP

> **适用项目**：DC-Agent  
> **起草日期**：2026-06-01  
> **来源**：[DC-Agent升级计划_CodexSkills_2026-06-01.md](DC-Agent升级计划_CodexSkills_2026-06-01.md)

这份 SOP 用来约束 Codex 参与 DC-Agent 日常开发、排障、前端验证和交付的基本流程。核心原则是：先确认问题，再执行；先保留证据，再判断；完成前必须验证。

---

## 一、任务入口判断

每次开工前，先把任务归入下面任一类型。

| 类型 | 典型输入 | 优先使用的 Skill | 产出 |
|---|---|---|---|
| 需求不清 | “我想做一个功能”“这里体验不好” | `grill-me` / `to-prd` | 关键问题、推荐答案、PRD |
| 任务拆分 | PRD、长计划、多条反馈 | `triage` / `to-issues` | 可执行任务清单 |
| Bug / 故障 | 启动失败、接口不通、日志报错 | `diagnose` / `systematic-debugging` | 证据、假设、根因、修复 |
| 核心逻辑开发 | `router/`、`dc_engines/`、`harness/` | `tdd` / `test-driven-development` | 先失败后通过的测试 |
| WebUI 修改 | `dashboard/`、`hermes-webui/` | `frontend-design` / `playwright` | UI 改动和浏览器验证 |
| 架构治理 | 模块边界、重复逻辑、扩展困难 | `improve-codebase-architecture` | 问题清单和重构优先级 |
| 收尾交接 | 修改完成、准备提交或交给下一轮 | `verification-before-completion` / `handoff` | 验证记录、风险、下一步 |

## 二、标准执行流程

### Step 1：确认范围

开始前必须确认：

- 本次任务的目标和非目标。
- 预计涉及的目录。
- 是否会触碰敏感目录：`data/`、`logs/`、`hermes-config/`。
- 是否需要启动本地服务或访问外部账号。
- 当前工作区是否已有无关改动。

如果工作区已有无关改动，只处理本次任务需要的文件，不回滚、不格式化无关文件。

### Step 2：选择工作模式

按任务类型选择最小必要 skill：

- 功能方案还没定：先用 `grill-me`，不要直接改代码。
- 需要写成产品需求：用 `to-prd`。
- 已有计划要落地：用 `executing-plans`。
- 故障排查：先用 `diagnose`，复杂链路再用 `systematic-debugging`。
- 核心业务逻辑：优先用 `tdd`。
- 前端体验：先用 `frontend-design`，落地后用 `playwright` 验证。
- 完成前：用 `verification-before-completion`。

### Step 3：执行实现

实现过程中遵守项目规则：

- Python 路径处理优先使用 `pathlib.Path`。
- 新增注释统一使用英文。
- 不新增 `xxx_SUMMARY.md` 这类报告文件。
- 不把 token、cookie、数据库、上传文件、运行日志提交到 git。
- 不对无关文件做大范围重排、格式化或重构。
- WebUI 修改必须维护组件化，避免复制粘贴式重复代码。

### Step 4：验证

按改动范围选择验证命令，不要求每次全部执行。

| 改动范围 | 建议验证 |
|---|---|
| Python 代码 | `ruff format .`、`ruff check .`、相关 `pytest` |
| 核心逻辑 | 对应单元测试或集成测试 |
| 启动链路 | `uv run main.py` 或相关启动脚本 |
| Dashboard | `cd dashboard && pnpm dev`，再做浏览器验证 |
| Hermes WebUI | `cd hermes-webui && pnpm dev`，再做浏览器验证 |
| 文档 | 检查链接、命令、路径是否仍然存在 |
| 提交前 | `make clean-pyc`、`make check-clean` |

无法执行的验证必须说明原因，并给出替代检查。

### Step 5：收尾记录

每个较大改动收尾时记录：

- 背景：为什么做。
- 方案：改了什么。
- 涉及目录：哪些文件或模块。
- 验证：实际跑过哪些命令，结果如何。
- 风险：未覆盖的场景或依赖条件。
- 回滚：如何恢复到修改前状态。

## 三、排障模板

排障时把过程沉淀到 [运维排障记录.md](运维排障记录.md) 或对应专题 SOP。

~~~markdown
## YYYY-MM-DD | 问题标题

### 现象
- 用户看到什么。
- 日志或接口返回什么。

### 证据
```bash
# 实际执行过的检查命令
```

### 根因
- 根因 1。
- 如有多个问题，拆开写。

### 修复
```bash
# 修复命令或关键变更
```

### 验证
```bash
# 验证命令
```

### 经验
- 后续遇到同类问题如何判断。
~~~

## 四、PR / 提交前 checklist

提交或交接前确认：

- [ ] 只包含本次任务相关改动。
- [ ] 没有提交 `data/temp/`、`data/output/`、数据库、token、日志或本地备份。
- [ ] 涉及 `data/config/` 的内容已确认不含 secret。
- [ ] 已运行必要验证，或明确说明未运行原因。
- [ ] Python 改动已按需运行 `ruff format .` 和 `ruff check .`。
- [ ] 提交前已运行 `make clean-pyc` 和 `make check-clean`。
- [ ] PR 标题和描述使用英文，标题符合 conventional commits。

## 五、常用提示词

```text
用 grill-me 拷问我这个 DC-Agent 功能方案。请先阅读相关代码和 DOC，再一次问一个关键问题，并给出你的推荐答案。
```

```text
用 diagnose 排查 DC-Agent 这个问题。请先看日志、启动脚本、配置，再给出根因假设和验证步骤。
```

```text
用 tdd 改这个核心逻辑。请先补最小失败测试，再实现，再跑相关测试。
```

```text
用 frontend-design 优化 dashboard 这个页面，重点看运维可读性、状态表达、操作路径和移动端布局。
```

```text
用 verification-before-completion 检查这次 DC-Agent 修改是否完成。请确认测试、lint、运行验证、文档和敏感文件状态。
```
