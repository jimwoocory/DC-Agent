# DC-Agent 飞书卡片类型设计清单

> 目标：统一小助手、Router、Harness、Hermes、知识库、媒体生成和员工入职流程里的飞书卡片体验。  
> 设计原则：温柔、可信、可追溯、老板能快速看懂、员工能轻松补齐材料。

## 1. 设计总原则

### 1.1 真实性状态必须显性展示

所有涉及公司事实、客户、员工、项目、文件、预算、数据、进度、工具执行结果的卡片，都需要展示真实性状态。

建议状态：

| 状态 | 展示文案 | 适用场景 |
|---|---|---|
| 已核验 | 已基于真实资料生成 | 用户提供了原文、附件、飞书链接、知识库或 Harness 资料 |
| 部分待确认 | 已基于现有资料整理，仍有信息待确认 | 有资料但缺少关键字段 |
| 需要补充 | 为避免内容失真，需要补充资料 | 请求涉及真实公司内容但没有来源 |
| 示例/假设 | 以下为示例内容，需替换为真实信息 | 用户明确要求模板、示例、模拟 |
| 执行失败 | 任务执行失败，已给出诊断建议 | Hermes、Claude CLI、Codex、媒体生成失败 |

### 1.2 语气原则

- 小助手表达要温柔、耐心、体贴。
- 真实性约束要坚定，但不要像审计、风控或质问。
- 缺资料时使用关怀式表达，例如：
  - “我先帮你把这件事稳住。”
  - “为了不让内容失真，我还需要一点可核验的资料。”
  - “有多少给多少，我会接着往下处理。”
- 对老板要更简洁稳重，对员工可以多一点鼓励和减压。

### 1.3 每张卡片建议统一包含

| 区块 | 是否必选 | 说明 |
|---|---|---|
| 标题 | 必选 | 让用户一眼知道卡片类型 |
| 状态 | 必选 | 处理中、已完成、需要补充、失败等 |
| 真实性状态 | 公司相关任务必选 | 已核验、部分待确认、需要补充、示例/假设 |
| 来源 | 公司相关任务必选 | 附件、飞书链接、知识库、Harness task、CLI 输出 |
| 主体内容 | 必选 | 回复、草稿、结果、错误说明 |
| 下一步 | 建议必选 | 补材料、确认、继续生成、交给老板审核 |
| 技术信息 | 可选 | provider、prompt_id、task_id、耗时、队列位置 |

## 2. 建议视觉母版

不要每个业务场景单独做完全不同的卡片，建议收敛为 6 个母版。

| 母版 | 覆盖场景 | 设计重点 |
|---|---|---|
| 轻量回复卡 | 闲聊、普通问答、短总结 | 温柔、轻巧、少技术信息 |
| 材料收集卡 | 缺资料、补资料、资料已接收、待确认 | 明确缺什么，员工容易补 |
| 结构化交付卡 | 邮件、公告、文案、方案、报告 | 可直接复制、分段清楚、待确认项明显 |
| 媒体生成卡 | 生图、图生视频、文生视频 | 展示 prompt、比例、风格、状态、结果 |
| 深度任务卡 | Hermes 排队、进度、完成、失败 | 队列、阶段、耗时、结果、诊断 |
| 系统/知识库卡 | DevOps、模型状态、知识库沉淀、来源追踪 | 稳重、可审计、便于老板看 |

## 4. 需要新增或统一升级的业务卡片

### 4.1 真实性资料请求卡

用于：员工提出公司真实任务，但没有提供真实依据。

触发场景：

- 公司公告
- 员工通知
- 客户邮件
- 品牌文案
- 项目分析
- 图片/视频素材生成
- Hermes 深度分析

建议字段：

| 字段 | 示例 |
|---|---|
| 标题 | 需要一点真实资料 |
| 温柔提示 | 我先帮你把这件事稳住。为了不让内容失真，我还需要一点可核验的资料。 |
| 需要补充 | 背景、目标对象、原文/附件、必须出现/不能出现的信息 |
| Harness 状态 | `truth_intake / blocked` |
| task_id | `#abc12345` |
| 操作建议 | 直接发文件、图片、截图、飞书链接或粘贴原文 |

### 4.2 资料已接收卡

用于：员工补充真实材料后。

建议字段：

| 字段 | 示例 |
|---|---|
| 标题 | 资料已收到 |
| 状态 | 已归档，正在继续处理 |
| 资料来源 | 附件 2 个、飞书链接 1 个、文字材料 1 段 |
| 归档位置 | `data/harness_intake/raw/...` |
| 下一步 | 已交回 Router / Hermes 继续执行 |

### 4.3 来源追踪卡

用于：所有公司事实类输出之后，展示“这次回答基于什么”。

建议字段：

| 字段 | 示例 |
|---|---|
| 真实性状态 | 已基于真实资料生成 |
| 来源 | 附件、飞书文档、知识库、Harness task |
| 知识库 | 品牌规范、营销素材、中台运营 |
| 未确认项 | 预算、发布时间、客户审批意见 |
| 风险提示 | 以下内容仍需老板确认 |

### 4.4 邮件草稿卡

### 4.5 普通文案卡

建议结构：

### 4.6 生图任务卡

用于：结果不完整或需要员工补信息。

建议字段：

|---|---|
| 标题 | 还差一点信息 |
| 温柔提示 | 我已经先整理到这一步了，还需要你确认几项 |
| 待确认项 | 时间、对象、预算、发布渠道 |
| 当前可用内容 | 已经生成的草稿或摘要 |
| 下一步 | 补充后继续生成 |

### 4.18 Skill 运维交互卡

用于：会话 / 群聊 / 老板或同事问答沉淀为 `.skill` 后，在飞书内完成列表查看、详情检查、质量审阅、回收站恢复和危险操作二次确认。

第一批卡片：

| 卡片 | 用途 |
|---|---|
| Skill 列表卡 | 展示老板 / 同事 Skill，提供查看、审阅、纠错、回滚、删除 |
| Skill 详情卡 | 展示 slug、version、样本量、知识源、文件清单 |
| Skill 质量审阅卡 | 展示 score、通过项、风险项和后续操作 |
| 已删除 Skill 回收站卡 | 展示软删除 Skill，提供恢复和路径查看 |
| 危险操作确认卡 | 删除 / 回滚前二次确认，后端继续校验管理员权限 |

按钮 value 约定：

```json
[
  {
    "source": "hermes_skill_card",
    "action": "inspect",
    "kind": "boss",
    "slug": "yang-zong",
    "version": "20260521T120000Z"
  },
  {
    "source": "hermes_skill_card",
    "action": "delete_confirm",
    "kind": "boss",
    "slug": "yang-zong"
  }
]
```

注意：`delete` / `rollback` 属于高风险动作，列表 / 详情卡只能发起 `*_request`，必须先进入确认卡；确认按钮仍然要在后端校验管理员权限。

## 5. 卡片状态颜色建议

| 状态 | 颜色建议 | 说明 |
|---|---|---|
| 正常完成 | 绿色 | 已完成、可使用 |
| 处理中 | 蓝色 | 思考中、生成中、分析中 |
| 需要补充 | 红色 | 缺材料、待确认 |
| 失败/异常 | 红色 | 执行失败、工具异常 |
| 示例/草稿 | 灰色或紫色 | 非真实结论，需替换 |

### 5.1 Feishu 正文颜色渲染口径

2026-05-21 实测结论：

- 卡片 header 可以用飞书 `template` 色板，缺资料 / 待确认 / 警告类标题统一用 `red` 模板；完成类标题用 `green` 模板。
- 卡片正文里的 `lark_md` `<font color='...'>` 不要使用 hex 色值。`#940000` / `#008000` 这类自定义色在真实飞书卡片里会被吞掉，导致正文标红、标绿消失。
- 正文状态值必须用飞书稳定支持的命名色：警告、待确认、需要补充用 `<font color='red'>...</font>`；正常、已归档、已入库用 `<font color='green'>...</font>`；弱化说明用 `<font color='grey'>...</font>`。
- 如果想表达更深的“警告红”，只能通过 header 红色模板、加粗、状态点和文案层级增强，不要再把正文改成 hex。
- 这条规则同时适用于 Codex、Claude Code 和后续知识库训练材料，避免新 session 重新把正文红绿改没。

## 6. 卡片与 Router / Harness / 知识库的关系

### 6.1 Router 应输出卡片语义

Router 决策建议带上：

```json
{
  "intent": "ops_writing",
  "provider_id": "aihubmix/gemini-3-flash-preview",
  "prompt_id": "employee_email_reply",
  "card_type": "email_draft",
  "truth_policy": "strict",
  "kb_names": ["中台运营", "品牌规范"]
}
```

### 6.2 Harness task 应记录卡片上下文

Harness payload 建议记录：

```json
{
  "task_id": "xxx",
  "domain": "truth_intake",
  "card_type": "material_request",
  "prompt_id": "announcement_copy",
  "prompt_version": "2026-05-20.1",
  "source_material_archive": "data/harness_intake/raw/...",
  "kb_names": ["中台运营", "营销素材"],
  "truth_status": "need_materials"
}
```

### 6.3 知识库沉淀应回显到卡片

当资料被归档或入库，卡片要让用户知道：

- 收到了哪些资料
- 是否已经入库
- 入到了哪个知识库
- 是否还需要人工审核
- 后续回答会不会引用它
